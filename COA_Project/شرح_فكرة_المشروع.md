# شرح فكرة مشروع C.O.A بالكامل

## الهوية (للجمهور التقني)

**C.O.A — Council of Agents، منصة الدفاع السيبراني السيادية للبيئات الحساسة.**

| | |
|--|--|
| **Tagline (عربي)** | ذكاء دفاعي محلي. بلا سحابة. بلا تسريب. |
| **Tagline (English)** | Sovereign Defense Intelligence. On-Premise. Air-Gapped Ready. |

الرسالة في سياق عرض أو هاكاثون دفاعي: المشروع **لا يُقاس كبديل مباشر** لـ CrowdStrike أو SentinelOne (سحابة + телеметري خارجي)، بل كحل لمعضلة **السيادة الرقمية + التحليل المتقدم** في بيئات لا تقبل إرسال بيانات حساسة خارج الحدود أو الشبكة الداخلية.

## بيان المشكلة

الجهات الدفاعية والمنشآت الحساسة في المملكة تواجه معضلة: الحلول الأمنية المتقدمة تعتمد على السحابة وترسل بيانات حساسة لخوادم خارج الحدود، بينما الحلول المحلية التقليدية تفتقر للذكاء الاصطناعي وقدرات التحليل المتقدمة. النتيجة: إما التضحية بالسيادة الرقمية، أو التضحية بجودة الكشف. **C.O.A يحل هذه المعضلة** بجمع التحليل والتقارير الدفاعية (وكلاء، MITRE، YARA، تقارير حوادث) في مسار يمكن تشغيله بالكامل داخل النطاق المحلي أو المعزول، مع إبقاء التكاملات السحابية **اختيارية** فقط عند السياسة الأمنية للجهة.

---

## الجمهور المستهدف

- مراكز العمليات الأمنية (SOC) في الجهات الدفاعية.
- المنشآت الحرجة: طاقة، اتصالات، مياه، نقل.
- الوحدات ذات الأنظمة **المعزولة (Air-Gapped)**.
- شركات الصناعات الدفاعية.

**سياق الهاكاثون:** تقديم **الشركة الوطنية للأنظمة الميكانيكية (SAMI)** كشريك في منظومة الدفاع الوطني يبرز فهم السياق المحلي؛ يمكن وصف المنصة كـ **جهة مستفيدة محتملة** لسيناريوهات التدقيق على الأصول المحلية (مع بقاء التبني مسألة اختبار وتنظيم).

## الركائز الثلاث (ومواءمة ما يثبته المشروع اليوم)

### 1) السيادة (Sovereignty)

- جمع البيانات والتحليل وتوليد التقارير محلياً؛ **Ollama** للنموذج اللغوي دون إلزام بمزود سحابي للذكاء الاصطناعي.
- **VirusTotal** اختياري: مفيد في بيئات **مسموح** فيها الاتصال الخارجي؛ في البيئة السيادية أو المعزولة **يُعطّل** حتى لا تخرج مؤشرات إلى خدمات خارجية — وهذا يحافظ على اتساق عبارة «بلا تسريب» مع الواقع التقني.

### 2) التخصص الدفاعي (Defense-Grade Specialization)

- ربط **MITRE ATT&CK** وتقارير حوادث منظمة عبر وكيل **Incident Reporter**.
- **YARA** مع قواعد أساسية في المستودع؛ الصياغة الدفاعية القوية تتضمن **قابلية التوسيع** بقواعد مخصصة لتهديدات مستهدفة للمنطقة (خارطة طريق/تخصيص تشغيلي).
- دعم بروتوكولات أنظمة صناعية (ICS/OT) **ضمن التوسع المستقبلي** — الإصدار الحالي يركز على مؤشرات نظام التشغيل الشائعة (عمليات، شبكة، إلخ).

### 3) الجاهزية التشغيلية (Operational Readiness)

- إمكان التشغيل **Air-Gapped** عند توفير التبعيات والنماذج مسبقاً داخل الشبكة.
- **Dry-run** والتحقق من الأوامر ومسار موافقات قبل الإجراءات الحساسة.
- **ثنائية اللغة:** وثائق وواجهات عربية مع مخرجات تقنية إنجليزية في التقارير؛ دمج تقرير رسمي AR+EN في ملف تصدير واحد يمكن اعتباره خطوة تالية إذا طلبها العرض.

## كيف يعمل منطقياً؟ (ملخص تقني)

1. **جمع البيانات** — قراءة حالة النظام (عمليات، شبكة، إلخ).
2. **تحليل التهديدات** — تسجيل وتصنيف ما يبدو مشبوهاً.
3. **تعزيزات اختيارية** — VirusTotal (عند التفعيل)، YARA.
4. **وكلاء (CrewAI)** — أدوار متخصصة بما فيها **تقرير الحوادث** وربط MITRE.

المشروع **محرك فحص** مع واجهات متعددة (React، Tkinter، CLI، Helpdesk API) وليس موقعاً تقليدياً متعدد المسارات.

## ماذا يرى المستخدم؟

| الطريقة | الفكرة |
|--------|--------|
| **واجهة React (SPA)** | صفحة واحدة + API محلي؛ تبويبات Threats / Processes / Network / Logs وتصدير تقارير. |
| **واجهة Tkinter** | سطح مكتب بنفس الفكرة تقريباً. |
| **CLI** | `main.py` مع خيارات مثل `--dry-run` و`--vt` و`--yara`. |
| **Helpdesk API** | REST لدمج الفحص مع بوتات الدعم. |

المخرجات في **`reports/`** و**`logs/`**.

---

## لماذا «مجلس وكلاء»؟

تقسيم مهمة الفحص إلى وكلاء متخصصين بدل سكربت أحادي، مع تقارير تصلح للإدارة وللفريق الفني.

| الوكيل | الدور |
|--------|--------|
| Data Collector | جمع بيانات النظام |
| Threat Hunter | تحليل وصيد التهديدات |
| Solution Advisor | اقتراح معالجة (dry-run والتحقق) |
| Incident Reporter | تقرير حادث، شدة، MITRE، ملخصات |

---

## ملخص

**C.O.A** منصة دفاع سيبراني **سيادية** تجمع الذكاء المحلي، تقارير الحوادث، وMITRE وYARA في فئة لا تتنافس مع EDR سحابي عام، بل تخدم **البيئات الحساسة والمعزولة** حيث تكون السيادة شرطاً.
